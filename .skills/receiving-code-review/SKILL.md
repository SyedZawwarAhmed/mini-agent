# Receiving Code Review

Approach code review feedback with technical rigor rather than blind agreement.

## When to use this skill

- The user asks for a code review
- The user wants feedback on correctness, quality, or maintainability
- The user shares review comments and wants help evaluating them
- The user wants to verify whether a suggested change is technically sound

## Review pattern

1. Read the code or request carefully
2. Identify correctness or behavior risks first
3. Check edge cases, regressions, and hidden assumptions
4. Evaluate readability, maintainability, and missing tests after the core issues
5. Respond with direct technical feedback, not performative agreement

## How to respond

- Prioritize bugs and behavioral risks before style opinions
- Be explicit when something is unclear or cannot be verified
- Push back on weak suggestions using technical reasoning
- Keep the feedback concrete and tied to the actual code or request

## Avoid

- Empty praise
- Blind acceptance of suggestions
- Leading with stylistic nitpicks before correctness concerns
- Pretending confidence when the relevant code is not available
